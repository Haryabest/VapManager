# AGV New UI — Руководство для разработчиков

## Назначение проекта

Qt Widgets-приложение для управления AGV, пользователями, задачами ТО, уведомлениями и чатами по задачам.  
Ключевая идея: все клиенты работают с единой MySQL БД, UI обновляется через сигналы `DataBus`.

## Архитектурная схема

- `main.cpp` — старт приложения, инициализация подключения, запуск окна/логина.
- `mainwindow.*` — контейнер верхнего уровня.
- `leftmenu.*` — основная рабочая оболочка (страницы, навигация, календарь, уведомления, профиль, логи, чаты).
- `db*.{h,cpp}` — доступ к данным (пользователи, модели, задачи, чаты, уведомления).
- `*page.*` / `*dialog.*` / `*widget.*` — UI-модули по функциям.
- `databus.*` — центральная шина сигналов между модулями.
- SQL-скрипты — создание/обновление схемы БД.

## Разбор ключевых методов (`void`) и почему они так сделаны

Ниже перечислены основные `void`-методы, влияющие на поведение системы.

### `leftmenu`
- `initUI()` — собирает весь основной экран единожды; так снижается риск рассинхрона между вкладками.
- `showCalendar()/showAgvList()/showModelList()/showLogs()/showUsersPage()/showProfilePage()/showChatsPage()` — централизованный роутинг по страницам.
- `hideAllPages()` — гарантирует, что одновременно видна только одна основная страница.
- `setSelectedMonthYear()` — полная пересборка календарной таблицы для выбранного периода.
- `selectDay()` — выделение конкретной даты после фильтра.
- `updateUpcomingMaintenance()` — формирует правую ленту ТО.
- `updateSystemStatus()` — пересчет агрегированных счетчиков.
- `updateAgvCounter()` — актуальный счетчик AGV.
- `updateNotifBadge()` — актуальный счетчик непрочитанных уведомлений.
- `reloadLogs(int maxRows)` — загрузка лога с фильтрами и ограничением строк.
- `changeAvatar()` — смена аватара текущего пользователя.
- `showNotificationsPanel()` — панель уведомлений с обработкой переходов в чат/раздел.
- `reloadChatsPageList()` — быстрый рефреш списка чатов.

Почему так:
- Тяжелые перерисовки делаются по событию (`DataBus`) и таймерам, чтобы UI был отзывчивым.
- Отложенные обновления (`pendingCalendarReload_`) используются, чтобы не сбрасывать пользователя с активной вкладки.

### `TaskChatDialog`
- `setupCreateUi()` — форма создания чата.
- `setupViewUi()` — форма просмотра треда.
- `sendNewThread()` — создание треда + первое сообщение.
- `sendReply()` — отправка ответа.
- `refreshMessages()` — периодическая подгрузка сообщений, smart-autoscroll.
- `closeThreadByAdmin()` — завершение треда админом/техом.
- `deleteThreadByAdmin()/deleteThreadForAll()` — физическое удаление треда.
- `deleteThreadForMe()` — soft-hide у конкретного пользователя.

Почему так:
- Разделение "закрыть" и "удалить" позволяет сохранить историю и при этом остановить переписку.
- Polling таймер в чате дает почти "мгновенное" обновление без отдельного websocket-сервера.

### `UsersPage`
- `loadUsers()` — единая точка загрузки списка пользователей.
- `updateUserInList(const QString&)` — локальное обновление строки пользователя.

Почему так:
- Изоляция загрузки в одном месте уменьшает риск reentrant-багов.

### `AgvSettingsPage`
- `loadAgv(const QString&)` — загрузка карточки AGV.
- `highlightTask(const QString&)` — фокус на задаче из правого блока ТО.
- `clearTasks()/buildTasksHeader()/addTaskRow()` — управление таблицей задач.
- `openAddTaskForm()/openEditTaskForm()` — формы редактирования задач.
- `deleteTask()/deleteSelectedTasks()/restoreDeletedTasks()` — удаление и undo.
- `toggleEditMode()/updateCheckboxVisibility()` — массовый режим редактирования.
- `enterAgvEditMode()/leaveAgvEditMode()/refreshAgvEditButtons()` — режим редактирования AGV.
- `saveAgvInfo()` — сохранение данных AGV.
- `openTaskHistoryDialog()` — просмотр истории обслуживания.

Почему так:
- Задачи и карточка AGV разделены на подрежимы, чтобы снизить риск случайных правок.

### `ListAgvInfo`
- `addAgv()/removeAgvById()` — локальные изменения списка.
- `loadAgvList()` — чтение AGV из БД.
- `rebuildList(const QVector<AgvInfo>&)` — перерисовка списка.
- `applyFilter(const FilterSettings&)` — фильтры и сортировки.
- `rebuildShownChunk()` — порционная отрисовка для больших списков.
- `showUndoToast()/restoreDeletedAgvs()` — undo удалений.

Почему так:
- Chunk-рендер помогает работать с большими списками AGV без фризов UI.

### `ModelListPage` / `TemplatePageWidget`
- `openAddModelDialog()/openDeleteDialog()` — операции над моделями.
- `reloadFromDatabase()` — синхронизация с БД.
- `showListMode()/showTemplateMode()/showModelDetailsMode()` — переключение внутренних экранов.
- `TemplatePageWidget::onSaveClicked()/toggleDeleteMode()/deleteSelected()/addRow()` — редактирование шаблона задач.

Почему так:
- Моды (list/template/details) упрощают состояние интерфейса.

### `DataBus`
- `triggerNotificationsChanged()` + сигналы `agvListChanged/modelsChanged/calendarChanged/userDataChanged/notificationsChanged`.

Почему так:
- Слабая связанность модулей: источник данных не знает, кто именно подписан на обновление.

### `UiActionLogger`
- `attachExistingActions()/attachAction()` — подписка на UI-action объекты.
- `writeUiLog()` — запись в файл логов.
- `compactText()/resolveWindowName()` — нормализация текста и источника.

Почему так:
- Унификация UI-логов упрощает аудит и расследование багов.

## Поток данных: "делегирование -> уведомление -> чат"

1. Из карточки AGV/задачи формируется действие назначения/делегирования.  
2. Запись сохраняется в БД (`db_agv_tasks`, связанные таблицы).  
3. Создается уведомление через `addNotificationForUser`.  
4. `DataBus::notificationsChanged` обновляет badge/панель.  
5. Пользователь кликом открывает чат (`TaskChatDialog`) или создает новый.  
6. Сообщения и статус треда читаются/пишутся через `db_task_chat`.

## Почему поддерживается 500+ клиентов

- Клиенты stateless относительно бизнес-данных (источник истины — MySQL).
- Неблокирующие UI-обновления (таймеры + DataBus + отложенные перерисовки).
- Индексы по уведомлениям/чатам для быстрых выборок.
- UTF-8 (`utf8mb4`) на уровне таблиц и соединения.

## Правила доработки

- Не редактировать `debug/`, `release/`, `moc_*`, `qrc_*`.
- Новые поля БД — отдельным SQL-миграционным скриптом (и при необходимости обновить `install_agv_manager_db.sql`).
- После любых изменений чатов/уведомлений проверять:
  - создание/чтение/закрытие треда;
  - unread badge;
  - права по ролям (admin/tech/viewer).
- После правок UI в `leftmenu` всегда проверять, что вкладки не "перекрываются" друг с другом.

## Минимальный smoke-test после изменений

1. Логин 2 пользователями (`admin` и обычный пользователь).  
2. Назначить AGV -> проверить уведомление у пользователя.  
3. Открыть чат из уведомления -> отправить сообщения в обе стороны.  
4. Завершить чат админом -> проверить блокировку ввода у пользователя.  
5. Изменить профиль (русский ФИО) -> сохранить -> открыть список пользователей.  
6. В календаре выбрать неделю в настройках -> уйти на другую вкладку -> дождаться авто-сброса -> убедиться, что не перекинуло принудительно на календарь.
